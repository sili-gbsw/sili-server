# v1 기능 명세서

## 스팟용접 품질 판정 시스템 (Spot Welding Quality Gate)

**프로젝트 코드명**: `sili-server`
**문서 버전**: v1.0
**작성 기준**: `v1_아이디어_구체화.md`
**대상 시스템**: 백엔드 API 서버 (FastAPI 기반)

---

## 1. 문서 개요

### 1.1 목적
본 문서는 「이상 의심 타점이 발생한 부품만 자동으로 골라내는 스팟용접 품질 관문」 시스템의 백엔드 서버(`sili-server`)에 대한 기능 명세를 정의한다.

### 1.2 범위
- 스팟용접기 PLC로부터 수집된 타점 데이터의 수신·저장
- 재질·두께·품질등급·전극형상 기반 동적 판정 엔진
- 이상 점수화(0~100점) 및 상태 분기 (🟢 정상 / 🟡 주의 / 🔴 재검권장)
- 부품 마스터·재검 큐·학습 환류 관리
- 작업자 인터페이스용 REST API 제공

### 1.3 시스템 환경
| 항목 | 내용 |
| --- | --- |
| 언어/프레임워크 | Python 3.11+ / FastAPI |
| DB | MongoDB (Motor 비동기 드라이버 + Beanie ODM 권장) |
| API 버전 | `/api/v1` |
| 응답 포맷 | `ApiResponse[T]` (success/error 공통) |
| 인증 | (v1 범위 외 — 추후 확장) |

## 3. 도메인 모델

### 3.1 핵심 엔티티

| 엔티티 | 설명 | 주요 속성 |
| --- | --- | --- |
| **Part** (부품 마스터) | 차종/부품별 기준 정보 | part_id, material_code, t1, t2, quality_class, electrode_shape |

| **MaterialProfile** | 재질별 보정 계수 (config) | code, current_factor, time_factor, force_factor |
| **ThicknessLimit** | 두께 조합별 기준값 (config) | combo, I_min/max, T_min/max, F_min/max |
| **QualityClassTolerance** | 품질등급별 허용 편차 (config) | class, thin, thick |
| **NormalRangeLearning** | 라인별 자동 학습 정상 범위 | line_id, part_id, mean·std (전류/시간/가압력), sample_count |

## 4. 기능 목록 (Functional Requirements)

### F-01. 공정 데이터 수집 (Ingestion)

| 항목 | 내용 |
| --- | --- |
| **설명** | PLC 게이트웨이가 타점별 측정 데이터를 송신하면 검증 후 저장한다. |
| **트리거** | PLC가 타점 완료 시점에 HTTP POST 호출 |
| **입력 필드** | 전류(kA), 통전시간(cycle), 가압력(kN), 누적타수, 두께(t1·t2), 재질코드, 부품 ID, 포인트 ID, 타임스탬프 |
| **처리** | ① 필수 필드 검증 → ② 부품 마스터 조회 → ③ WeldEvent 저장 → ④ 판정 엔진(F-05) 호출 |
| **출력** | `WeldEvent` 레코드 + 즉시 판정 결과 |
| **API** | `POST /api/v1/weld-events` |
| **에러** | 부품 ID 미등록 시 → 🔴 재검권장 강제 분기 + 4xx 반환 |

### F-02. 부품 마스터 관리 (Part Master)

| 항목 | 내용 |
| --- | --- |
| **설명** | 차종·부품별 재질·두께·품질등급·전극형상을 사전 등록·관리한다. |
| **사용자** | 품질 관리자 |
| **처리** | CRUD + 부품 바코드 ↔ 마스터 매핑 |
| **검증** | material_code ∈ {MILD, HSLA, DP600, DP980, UHSS, GA, GI} / quality_class ∈ {A,B,C} / electrode_shape ∈ {C-TYPE, R-TYPE} |
| **API** | `GET/POST/PATCH/DELETE /api/v1/parts`, `GET /api/v1/parts/{part_id}` |
| **연동** | 부품 바코드 스캔 시 자동 조회 → F-01 데이터에 결합 |

### F-03. 정상 범위 자동 학습

| 항목 | 내용 |
| --- | --- |
| **설명** | 작업 시작 후 초기 100타 데이터의 평균 ± 3σ를 자동 정상 범위로 설정한다. |
| **트리거** | 동일 (라인·부품) 조합 신규 시작 시 자동 / 수동 재학습 호출 |
| **처리** | ① 초기 100건 수집 → ② 전류/시간/가압력별 μ, σ 산출 → ③ NormalRangeLearning 저장 |
| **보정** | F-08의 재검 결과를 환류 받아 μ·σ 점진 갱신 |
| **API** | `POST /api/v1/learning/start`, `GET /api/v1/learning/{line_id}`, `POST /api/v1/learning/reset` |

### F-04. 강제 격상 사전 체크

판정 점수 계산 이전에 다음을 우선 검사하고, 해당 시 상태를 강제 격상한다.

| 조건 | 결과 | 메시지 |
| --- | --- | --- |
| 재질 코드 불일치 (등록 vs 실측) | 🟡 주의 격상 | "재질 확인 필요" |
| 미등록 재질 부품 투입 | 🔴 재검권장 강제 | "미등록 재질 — 부품 마스터 확인" |
| 두께 비 `max(t1,t2)/min(t1,t2) > 3.0` | 🟡 주의 격상 | "두께 비 1:3 초과 — 수동 파라미터 확인" |
| 전극 형상 불일치 (`min(t1,t2)<3.2`→C-TYPE / ≥3.2→R-TYPE) | 🟡 주의 격상 | "전극 형상 불일치" |

> **구현 위치**: 판정 엔진 진입부 (`app/services/judgement.py` 예정)
> **출력**: `Judgement.forced_reason` 컬럼에 사유 코드 저장

### F-05. 판정 엔진 (Scoring)

| 항목 | 내용 |
| --- | --- |
| **설명** | 이탈률을 가중합하여 0~100점 산출 후 상태를 분기한다. |
| **입력** | WeldEvent + 부품 마스터 + config(`thickness_limits`, `material_profiles`, `quality_class_tolerance`) |
| **알고리즘** | ① 얇은 쪽 두께 선택 → ② 등급별 허용 편차 로드 → ③ MILD 기준값에 재질 보정 계수 적용 → ④ 파라미터별 이탈률 정규화 → ⑤ 가중합 점수 산출 |

#### 점수 계산식 (의사코드 발췌)

```text
score = dev(current) * 0.30
      + dev(weld_time) * 0.20
      + dev(force)    * 0.20
      + wear_score(cumulative_hits) * 0.15
      + thickness_dev_score(t)      * 0.10
      + material_mismatch_score()   * 0.05
```

#### 상태 분기

| 점수 | 상태 | 처리 |
| --- | --- | --- |
| 0 ~ 30 | 🟢 정상 | 다음 공정 진행 |
| 31 ~ 60 | 🟡 주의 | 로그 기록 + 추세 관찰 (재검 큐 등록 X) |
| 61 ~ 100 | 🔴 재검권장 | 재검 큐 등록 + 즉시 알림 (F-07) |

#### 파라미터 가중치

| 파라미터 | 가중치 |
| --- | --- |
| 용접 전류 | 30% |
| 통전 시간 | 20% |
| 전극 가압력 | 20% |
| 전극 마모 | 15% |
| 판재 두께 | 10% |
| 재질 불일치 | 5% (이진 플래그) |

## 6. 데이터 모델 (MongoDB 컬렉션 스케치)

> MongoDB 기반. 상세 스키마는 Beanie `Document` 또는 Pydantic 모델로 `app/models/`에 구현한다. `_id`는 ObjectId 기본, 다만 도메인 식별자(part_id 등)에는 별도 인덱스를 건다.

### 6.1 컬렉션 개요

| 컬렉션 | 용도 | 주요 필드 | 인덱스 |
| --- | --- | --- | --- |
| `parts` | 부품 마스터 | `part_id`, `material_code`, `t1`, `t2`, `quality_class`, `electrode_shape`, `weld_points[]`(embed), `created_at`, `updated_at` | unique(`part_id`) |
| `weld_events` | 타점 이벤트 시계열 | `event_id`, `part_id`, `point_id`, `current_kA`, `weld_time_cycle`, `force_kN`, `cumulative_hits`, `t1`, `t2`, `material_code`, `electrode_shape`, `judgement`(embed), `timestamp` | `{part_id:1, timestamp:-1}`, `{timestamp:-1}` |
| `reinspection_queue` | 재검 큐 + 결과 | `queue_id`, `part_id`, `event_ids[]`, `status`, `reason`, `result`(embed: `is_defect`, `notes`, `inspector_id`, `inspected_at`), `created_at`, `closed_at` | `{status:1, created_at:-1}` |
| `electrodes` | 전극 이력 | `electrode_id`, `shape_code`, `install_ts`, `replace_ts`, `cumulative_hits` | `{shape_code:1}` |
| `normal_range_learning` | 라인·부품별 정상 범위 | `line_id`, `part_id`, `params`(embed: 전류·시간·가압력 각 `mean`/`std`/`sample_count`), `updated_at` | unique(`line_id`+`part_id`) |
| `config` | 시스템 동적 설정 (단일 도큐먼트) | `thickness_limits`, `material_profiles`, `quality_class_tolerance`, `electrode_shape_rule`, `thickness_ratio_limit`, `electrode_wear_limit`, `min_pitch_mm`, `min_lap_mm`, `version` | — |
| `config_audits` | 설정 변경 이력 | `key`, `old_value`, `new_value`, `changed_by`, `changed_at` | `{changed_at:-1}` |

> **임베드 vs 참조 가이드**
> - `parts.weld_points`는 **임베드**: 한 부품에 속한 타점 정의는 거의 같이 조회되며 양도 제한적이다.
> - `weld_events.judgement`는 **임베드**: 1:1 관계이고 같이 조회된다 (조인 비용 절감).
> - `reinspection_queue.result`도 **임베드** (1:1 환류).
> - `reinspection_queue.event_ids`는 **참조**: 큐에 다수의 이벤트가 연결될 수 있고, 이벤트는 별도 시계열 조회가 잦다.

### 6.3 인덱스·운영 정책

- **시계열 컬렉션 옵션**: `weld_events`는 MongoDB 5.0+의 *Time Series Collection* 으로 생성하면 압축·범위 쿼리 성능이 유리하다 (`timeField: "timestamp"`, `metaField: "part_id"`).
- **TTL 인덱스**: 운영 1년 보관 정책을 둔다면 `weld_events.timestamp`에 TTL 인덱스(`expireAfterSeconds: 31_536_000`) 부여.
- **트랜잭션**: 재검 결과 등록 → 학습 환류 갱신은 단일 도큐먼트 임베드로 1회 update에 끝나도록 설계되어 멀티 도큐먼트 트랜잭션이 거의 불필요하다. 필요 시 replica set 구성을 전제로 `client.start_session()` 사용.
- **ID 정책**: 도메인 식별자(`part_id`, `event_id`, `queue_id`)는 사람이 읽을 수 있는 문자열을 별도 유지하고 unique 인덱스를 건다. `_id`는 ObjectId.
